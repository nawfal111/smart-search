Abstract

Smart Search is a Visual Studio Code extension that adds AI-powered semantic code search to any project. It lets a developer type a natural language question — "where do we validate the payment subscription?" — and receive the exact function and specific line number that answers it, even when no keyword from the query appears in the code. The system indexes source code by splitting it into function-level chunks, generating plain English summaries via GPT, embedding those summaries alongside the code with Voyage AI's code-specialized model, and storing the resulting vectors in Pinecone. Search queries are embedded at query time and matched via cosine similarity. A second LLM call pinpoints the most relevant line within each matched function. This paper documents the complete system design, all engineering challenges encountered during development, and the solutions implemented. It serves as a technical record of this personal productivity project.

01 Motivation and Personal Context

The problem that prompted this project is one every developer working on a medium-to-large codebase encounters daily: you know a piece of logic exists somewhere, but you cannot remember what it is called.

In a typical web application backend — a Laravel PHP project with 400 files, or a Node.js API with 200 TypeScript modules — the built-in VS Code search (Ctrl+Shift+F) and grep are the primary navigation tools. Both require you to know a keyword. If you know the function is named checkSubscription, grep will find it instantly. But if you only know what it does — "checks whether a user's subscription is still active" — you are stuck. You must either browse files manually, ask a colleague, or guess at keywords and hope.

This was a recurring personal pain point. The specific triggers were:

  • Returning to a codebase after weeks away and needing to find logic that was written months earlier under an unclear name.
  • Inheriting or reviewing a project written by someone else with different naming conventions.
  • Looking for where a specific business rule is enforced in a large multi-module system.

The solution space was clear: semantic search — search by meaning, not by keyword. The challenge was building it in a way that was fast to use, cheap to run, integrated into the existing editor workflow, and smart enough to avoid redundant work (not re-embedding unchanged code on every session start).

Core Goal

Type a natural language question into VS Code. Get back the exact file, function, and line number — even when no word from your question appears in the code.

The split between TypeScript and Python is deliberate and meaningful — each layer does exactly what it is best at:

03 Component Deep Dive

3.1 — VS Code Extension Layer

The extension is written entirely in TypeScript and compiled to out/extension.js. VS Code loads this compiled file when the extension activates. The source is organized as follows:

src/ ├── extension.ts ← Entry point. Wires up all 8 lifecycle events. ├── config.ts ← getBackendUrl() — reads smartSearch.backendUrl setting ├── handlers/ │ ├── searchHandler.ts ← POSTs to /search, returns results to webview │ └── replaceHandler.ts ← VS Code WorkspaceEdit for single and bulk replace ├── indexer/ │ ├── workspaceIndexer.ts ← Two-level hash logic; drives all indexing flows │ ├── chunker.ts ← Splits code into function chunks (11 languages) │ ├── localIndex.ts ← Reads/writes .smart-search/index.json │ ├── projectId.ts ← Generates/reads UUID in .smart-search/project-id │ └── userId.ts ← MD5(git global email) for Pinecone namespace └── utils/ └── webviewManager.ts ← Inlines CSS+JS into HTML for the webview panel

extension.ts — The Entry Point

The activate() function is called once by VS Code when the extension starts. It registers 8 lifecycle behaviours, all of which are non-blocking:

  1. Status bar item — shows indexing progress at the bottom of VS Code ("⟳ Smart Search: indexing 3/47…")
  2. Backend health check — GETs /health on startup; shows a warning if the Python server is not running
  3. Workspace indexing — walks all files in the background on every VS Code open
  4. Save listener — re-indexes a single file every time it is saved
  5. Delete listener — removes deleted files' vectors from Pinecone immediately
  6. Rename listener — removes old path vectors, re-indexes file at new path
  7. Re-index command — "Smart Search: Re-index Workspace" wipes Pinecone + rebuilds from scratch
  8. Search panel command — "Smart Search" opens the webview UI panel

chunker.ts — The Code Splitter

The chunker takes a file's raw text and returns an array of Chunk objects. Each chunk represents one function, method, or class. It runs entirely in TypeScript — no network call, no Python, just string parsing. This is important for performance: chunking a 500-line file takes under 1 ms.

The Chunk interface:

{
  id:         "src/auth.php::verifyToken",  // unique: relativePath::name
  name:       "verifyToken",
  type:       "function",                  // "function" | "class" | "method" | "file"
  content:    "function verifyToken($tok) { ... }",
  start_line: 12,
  end_line:   38,
  file:       "src/auth.php",               // relative path (portable)
  language:   "php",
}

The chunk ID uses the relative path (not absolute) so it is portable across machines. Moving the project folder does not change the ID.

workspaceIndexer.ts — The Brain

This file orchestrates everything. It implements two levels of hash comparison to decide what needs re-embedding (detailed in Challenge C-02). It also handles four distinct entry points: full workspace index, single file re-index, file deletion, and file rename.

userId.ts — Stable Identity

Reads the globally configured git email (git config --global user.email) and returns its MD5 hash. This is used as the user portion of the Pinecone namespace. If git email is not configured, the extension shows a clear error and refuses to index — no silent fallback that would create a different namespace each time.

webviewManager.ts — The Frontend Loader

VS Code webview panels need a single self-contained HTML string. The frontend is split into three files for maintainability (index.html, styles.css, main.js). This module reads all three and inlines the CSS and JS into the HTML before handing it to VS Code:

// Replace placeholder comments in HTML with actual file contents
return html
  .replace("<!-- STYLES -->",  `<style>\n${css}\n</style>`)
  .replace("<!-- SCRIPTS -->", `<script>\n${js}\n</script>`);

3.2 — Python Backend Layer

The backend is a plain Python HTTP server using the standard library's http.server.HTTPServer. It has no framework dependency — no Flask, no FastAPI. This was a deliberate choice to keep the install footprint minimal (four pip packages: openai, pinecone, voyageai, python-dotenv).

The server exposes four endpoints:

EndpointMethodPurpose
GET /healthGETLiveness check — returns {"ok": true}
POST /searchPOSTNormal regex search or AI semantic search
POST /indexPOSTSummarize + embed + upsert new/changed chunks
POST /wipePOSTDelete all vectors in a Pinecone namespace

All endpoints support CORS (the do_OPTIONS handler returns the necessary headers) so the webview can call them directly if needed in future.

server.py — Route Dispatcher

The SearchHandler class inherits from BaseHTTPRequestHandler. Routing is handled with a simple dict of path → method:

routes = {
    "/search": self._handle_search,
    "/index":  self._handle_index,
    "/wipe":   self._handle_wipe,
}

The server is started with a configurable host and port, both read from environment variables with defaults:

host = os.getenv("BACKEND_HOST", "localhost")
port = int(os.getenv("BACKEND_PORT", "8000"))

embedder.py — Voyage AI Client

Wraps the Voyage AI SDK. Two public functions:

  • embed_text(text) — embeds a single search query. Uses input_type="query".
  • embed_chunks(chunks) — embeds a list of code chunks in one batched API call. Uses input_type="document".

Before embedding, each chunk's text is constructed as: label + summary + code. This three-layer text is what Voyage AI actually processes.

summarizer.py — GPT Summarizer

Takes a code chunk, sends it to gpt-4o-mini, and returns a 1–4 sentence plain English description. The prompt specifically asks GPT to cover what the function does, whether it reads or writes data, and whether it contains logging or error handling. The summary is prepended to the code before embedding.

line_locator.py — GPT Line Finder

After Pinecone returns a matched function, this module reads the function's lines from disk, builds a numbered listing, and asks GPT which specific line is most relevant to the search query. Returns {"line": N, "content": "..."}.

pinecone_client.py — Vector DB Client

Wraps the Pinecone SDK. Four operations: upsert_chunks, delete_chunks, query_chunks, wipe_namespace. All operations are scoped to a namespace so different users' vectors never interfere.

search.py — Normal Text Search

Walks the workspace directory tree using os.walk, applies a compiled regex to every line of every file, and returns matches with file path, line number, content, and character offsets. Skips ignored folders (node_modules, .git, etc.).

Single-File Re-index (on Save)

When the user saves a file, the same pipeline runs but scoped to just that one file. The indexSingleFile() function is called with the file path and its current content (VS Code provides the text directly, so no disk read is needed). The two-level hash check still applies — if neither the file hash nor any function hash changed, the function returns immediately with no network calls.

Re-index Workspace Command

The "Smart Search: Re-index Workspace" command does a hard reset:

  1. POSTs to /wipe — Pinecone deletes all vectors in the namespace
  2. Deletes .smart-search/index.json locally — clears all stored hashes
  3. Calls indexWorkspace() — treats every function as new, re-embeds everything

07 Engineering Challenges and Solutions

This section documents every significant challenge encountered during development and the solution implemented. These are the real problems that shaped the architecture.

C-01 Multi-Language Syntax Parsing

The Problem

To index at the function level, the chunker must split any source file into its individual functions and methods. Every supported language expresses a function differently, and more critically, every language uses a different mechanism to delimit where a function ends.

Three fundamentally different syntaxes had to be handled:

  • Brace languages (PHP, TypeScript, JavaScript, Java, Go, Rust, C#, Swift, Kotlin, C, C++) — a block begins with { and ends with the matching }. Counting brace depth is required because functions can be nested and because braces also appear in string literals and comments.
  • Python — no braces. A function body is defined purely by indentation. A def block ends when a non-blank line appears at the same or lower indentation level as the def keyword itself.
  • Ruby — uses the keyword end. Blocks are opened with def, class, module, if, do, while, etc. and closed with end. The depth must be tracked by counting opener and closer keyword occurrences.

Beyond the block-end problem, each language also has different syntax for declaring a function. PHP uses function name(. TypeScript uses function name( and const name = () => and const name = async function(. Java uses ReturnType methodName(. Go uses func name(. These all had to be matched with separate regex patterns.

The Solution: Three Block-End Algorithms + Language-Specific Patterns

Algorithm 1: findBraceEnd — for all C-style languages. Scans characters one by one from the start of the function, counting { as +1 and } as -1. When depth returns to 0 after having been opened, that is the end of the function.

function findBraceEnd(lines: string[], startIdx: number): number {
  let depth = 0, started = false;
  for (let i = startIdx; i < lines.length; i++) {
    for (const ch of lines[i]) {
      if (ch === "{") { depth++; started = true; }
      else if (ch === "}") {
        if (started) depth--;
        if (started && depth === 0) return i;
      }
    }
  }
  return lines.length - 1;
}

Algorithm 2: findPythonEnd — for Python. Records the indentation (number of leading spaces) of the def line. Scans forward, skipping blank lines. Returns when a non-blank line with indentation ≤ the header's indentation is found.

function findPythonEnd(lines: string[], startIdx: number): number {
  const headerIndent = lines[startIdx].match(/^(\s*)/)?.[1].length ?? 0;
  for (let i = startIdx + 1; i < lines.length; i++) {
    if (lines[i].trim() === "") continue; // blank lines are valid inside a block
    const indent = lines[i].match(/^(\s*)/)?.[1].length ?? 0;
    if (indent <= headerIndent) return i - 1;
  }
  return lines.length - 1;
}

Algorithm 3: findRubyEnd — for Ruby. Starts with depth 1 (the opening def). Increments for any opener keyword (def, class, do, if, etc.) and decrements for end. When depth reaches 0, the function is closed.

function findRubyEnd(lines: string[], startIdx: number): number {
  let depth = 1;
  const openers = /^\s*(?:def|class|module|do|if|unless|while|until|for|begin|case)\b/;
  const closer  = /^\s*end\b/;
  for (let i = startIdx + 1; i < lines.length; i++) {
    if (openers.test(lines[i])) depth++;
    if (closer.test(lines[i])) { depth--; if (depth === 0) return i; }
  }
  return lines.length - 1;
}

For the function-start patterns, each language has its own set of regex rules. For example, TypeScript needs to handle all of these forms:

// TypeScript patterns (simplified)
{ re: /^(?:export\s+)?(?:async\s+)?function\s+(\w+)/,     type: "function" },
{ re: /^(?:export\s+)?(?:const|let)\s+(\w+)\s*=.*=>/,     type: "function" },  // arrow
{ re: /^(?:export\s+)?(?:const|let)\s+(\w+)\s*=.*function/, type: "function" },
{ re: /^(?:export\s+)?class\s+(\w+)/,                       type: "class"    },
{ re: /^\s+(?:async\s+)?(\w+)\s*\([^)]*\)\s*\{/,           type: "method"   },

Class Exclusion

An additional problem discovered during testing: when class chunks are indexed, they contain the full body of all their methods combined. This produces a vector with very high cosine similarity to almost any query about functionality in that file, because the class vector "averages" all its methods. The result is that class chunks dominate the top-K results and push the actually-relevant individual method results down.

The fix is simple: after chunking, filter out type="class" before sending to the backend. Individual methods are already indexed separately. Classes add noise, not signal.

// workspaceIndexer.ts
const chunks = allChunks.filter((c: Chunk) => c.type !== "class");

Fallback for Unsupported Languages

If a file's language is not in the supported list (or the chunker finds no function declarations), the entire file is returned as one chunk of type "file". This ensures every indexable file gets some representation in Pinecone, even if it is less granular.

C-02 Hash-Based Incremental Indexing (No Re-embed on Every Run)

The Problem

The naive approach to indexing is to embed every function every time VS Code opens. For a 300-function project, this would mean 300 GPT calls + 300 Voyage AI calls every time the developer opens their editor — roughly $0.30 per startup and 15–30 seconds of waiting. This is completely unacceptable for a daily-use tool.

The key insight is that the vast majority of code does not change between sessions. If 295 out of 300 functions are identical to yesterday, there is no reason to re-embed them. The challenge is knowing which ones changed — and doing so without any network calls.

The Solution: Two-Level Local Hash Store

A JSON file (.smart-search/index.json) stores MD5 hashes of every indexed function. Before making any API call, the system computes hashes locally (sub-millisecond) and compares them against the stored values.

The comparison happens at two levels:

Level 1 — File hash (fast pre-filter): Hash the entire file content with MD5. If this hash matches the stored value, the file is completely unchanged — skip all its functions without even looking at them. This handles the common case of untouched files in O(1) per file.

Level 2 — Function hash (precise check): Only runs if the file hash changed (meaning at least one function changed). For each function extracted by the chunker, compute MD5(normalizeCode(content)) and compare to the stored value. Only functions with mismatched hashes are queued for re-embedding.

// workspaceIndexer.ts — Level 2 comparison
for (const chunk of chunks) {
  const funcHash = hashFunction(chunk.content); // MD5(normalizeCode(...))
  const existing = existingFunctions[chunk.name];

  if (!existing || existing.hash !== funcHash) {
    // changed or new — queue for re-embed
    if (existing) toDelete.push(existing.chunkId); // delete old vector first
    toEmbed.push({ ...chunk, language });
  } else {
    // unchanged — keep existing Pinecone vector
    newFunctions[chunk.name] = existing;
  }
}
// find deleted functions
for (const name of Object.keys(existingFunctions)) {
  if (!newChunkNames.has(name)) toDelete.push(existingFunctions[name].chunkId);
}
Real-World Impact

300-function project with 3 functions changed:
Without hashing: 300 GPT calls + 300 Voyage AI calls ≈ 15 seconds, ~$0.30
With hashing: 3 GPT calls + 1 Voyage AI batch call ≈ 0.5 seconds, ~$0.003
Savings: 99% of API cost and time on every subsequent run.

The hash store is keyed by relative file path (not absolute), so renaming the project's parent folder does not invalidate any stored hashes.

C-03 Formatting-Insensitive Code Normalization

The Problem

Developers frequently use auto-formatters: Prettier for JavaScript/TypeScript, Black or autopep8 for Python, gofmt for Go. These tools change whitespace (trailing spaces, blank lines, indentation style) without changing logic. If the hash comparison used raw file content, running a formatter would change the MD5 hash of every function it touched — triggering unnecessary re-embeds for changes that have no semantic meaning whatsoever.

Example: Black (Python formatter) adds a blank line after every function definition. On a 100-function Python file, this would trigger 100 re-embeds — costing ~$0.03 and 5 seconds for changes that mean nothing to search.

The Solution: normalizeCode()

A normalization step is applied to every function's content before hashing for comparison purposes. The normalization removes formatting-only differences:

function normalizeCode(content: string): string {
  return content
    .split("\n")
    .map(line => line.trimEnd())          // remove trailing spaces and tabs
    .filter(line => line.trim() !== "") // drop blank and whitespace-only lines
    .join("\n")
    .trim();                               // remove leading/trailing whitespace
}

This makes the following changes invisible to the hash system:

  • Adding or removing blank lines inside or around a function
  • Adding or removing trailing spaces on any line
  • Leading/trailing whitespace at the start or end of the function block

These changes do still produce a different hash (correctly triggering re-embed):

  • Changing a variable name
  • Adding, removing, or modifying a line of logic
  • Changing indentation in a way that alters Python block structure
  • Renaming the function itself

Importantly, normalizeCode() is used only for hash comparison. The original, un-normalized content is what gets embedded and stored. The displayed preview in search results shows the actual code as written.

C-04 Local Server and the Python Distribution Problem

The Problem

The AI features require a Python server (server.py). This works perfectly when the developer has Python installed on their machine. But several real-world scenarios break this assumption:

  • A collaborator who works primarily in JavaScript and has never installed Python.
  • The developer's second laptop where Python was not set up.
  • Windows machines where Python was installed but the Microsoft Store App Execution Alias intercepts the python and python3 commands before they reach the actual Python binary — causing "Python was not found" errors even when Python is installed.

The original design hardcoded the backend URL as http://localhost:8000, making it impossible to run the server anywhere other than the developer's own machine.

The Solution: Configurable Backend URL

The backend URL was extracted into a VS Code configuration setting. All backend calls go through getBackendUrl() which reads this setting:

// config.ts
export function getBackendUrl(): string {
  const config = vscode.workspace.getConfiguration("smartSearch");
  const url = config.get<string>("backendUrl", "http://localhost:8000");
  return url.replace(/\/$/, ""); // strip trailing slash
}

The setting is declared in package.json and appears in VS Code's Settings UI under "Smart Search → Backend URL". This enables:

  • Local mode (default): Developer runs python server.py on their machine. No configuration needed.
  • Remote mode: Set smartSearch.backendUrl to a remote server URL. The Python server runs on a cloud VM, a shared team server, or any machine with Python. Collaborators without Python can use the extension normally by pointing at the remote server.
Windows-Specific Issue
On Windows, Python can be installed but remain inaccessible because Microsoft Store App Execution Aliases intercept python.exe. Fix: Settings → Apps → Advanced app settings → App execution aliases → turn off "App Installer - python.exe" and "App Installer - python3.exe". Then open a new terminal.
C-05 Function Size and the 6 000-Character Cap

The Problem

Functions vary wildly in size. A simple utility function might be 5 lines. A large controller method might be 400 lines. Embedding extremely large functions creates multiple problems:

  • API token limits: Voyage AI has a maximum input length. Sending a 3000-line function verbatim would exceed it.
  • Embedding quality: A vector representing 3000 lines of mixed code is a blurry average. A query about one specific aspect of that function (e.g. "where do we log errors?") will match poorly because the vector's attention is spread across hundreds of unrelated operations.
  • GPT summarization: gpt-4o-mini cannot produce a useful 4-sentence summary of a 2000-line function. The summary would be too generic to be useful for search.
  • Cost: Embedding 10 000-character functions at scale would consume tokens rapidly.

The Solution: Caps at Each Layer

Multiple independent caps ensure no single layer is overwhelmed regardless of function size:

LayerCapReason
chunker.ts6 000 chars stored in ChunkLimits what's sent to backend
summarizer.py3 000 chars sent to GPTKeeps prompt focused; reduces token cost
embedder.py8 000 chars sent to Voyage AINear model input limit
pinecone_client.py1 000 chars for content fieldPinecone metadata per-field limit
pinecone_client.py500 chars for summary fieldKeeps result payload compact

When a function is longer than 6 000 characters, only the first 6 000 are embedded. The function signature, name, docstring, and opening logic — which carry the most semantic signal for matching queries — are almost always in the first portion of a function. Error handling, return statements, and edge cases that dominate long tails are less discriminative for retrieval purposes.

The MAX_CHUNK_CHARS constant in chunker.ts controls this limit and can be adjusted without changing any other part of the system.

C-06 Relevance Accuracy Threshold

The Problem

Pinecone's query(top_k=10) always returns 10 results — the 10 most similar vectors — regardless of how similar they actually are. If the project has only 2 functions genuinely related to the query, the remaining 8 results are the next-closest matches in the vector space, which might be completely unrelated code. Showing these irrelevant results is worse than showing nothing: it creates noise, wastes the developer's attention, and erodes trust in the tool.

The Solution: Configurable Similarity Threshold

After Pinecone returns the top-10 results, a filter is applied that keeps only results above a minimum cosine similarity score. The default threshold is 0.35 (35%).

# server.py — filter and sort by score
results = sorted(
    [r for r in all_results if r["score"] >= threshold],
    key=lambda r: r["score"],
    reverse=True,
)

The threshold is exposed in the UI as a 1–100 integer slider. The backend converts it by dividing by 100. If the user clears the field or enters an invalid value, the backend falls back to DEFAULT_AI_THRESHOLD defined in config.py. This means it is also easy to change the default system-wide.

In practice with voyage-code-2, thresholds work roughly as follows:

ThresholdBehaviour
20–30%Broad — many results including weak matches. Useful for exploration.
35% (default)Balanced — returns genuine matches for clear queries.
50–60%Strict — only very confident matches. May return fewer results.
70%+Very strict — only near-exact semantic matches.
C-07 Automatic .gitignore Management

The Problem

The .smart-search/ folder should never be committed to git. It contains:

  • project-id — if pushed, anyone who clones the repo gets the same project-id. Combined with their own git email hash, they write into a Pinecone namespace that looks like yours, creating confusion and overwriting vectors.
  • index.json — contains absolute or relative file paths and function hashes specific to one machine's state. These are meaningless to anyone else and could cause indexing to skip functions that are new to them.

Relying on developers to manually add the entry to .gitignore is unreliable — it will be forgotten.

The Solution: Auto-update .gitignore on Every Run

Every time getProjectId() is called (which happens on every indexing run), it calls addToGitignore(). This function checks if the entry already exists before writing, so it is safe to call repeatedly:

function addToGitignore(workspaceRoot: string): void {
  const entry   = ".smart-search/";
  const comment = "# Smart Search — local function hashes (not committed)";
  const gitignorePath = path.join(workspaceRoot, ".gitignore");

  if (fs.existsSync(gitignorePath)) {
    const content = fs.readFileSync(gitignorePath, "utf8");
    if (!content.includes(entry)) { // only append if not already there
      fs.appendFileSync(gitignorePath, `\n${comment}\n${entry}\n`);
    }
  } else {
    fs.writeFileSync(gitignorePath, `${comment}\n${entry}\n`); // create .gitignore
  }
}

The non-fatal catch block ensures that a read-only .gitignore or a permission error does not crash the extension — it logs a warning and continues.

C-08 Multi-User Namespace Isolation via Git Email

The Problem

Pinecone is a shared cloud database. If two developers work on the same repo (same project-id) and use the same Pinecone account, they need their vectors to be completely isolated. Otherwise:

  • Developer B's re-index of verifyToken() overwrites Developer A's vector for the same function. This is an error — their code states might differ and the vector should reflect each person's current working copy.
  • Developer A's search returns results shaped by B's embeddings, not their own.

Using a random ID per user would solve the isolation problem but breaks cross-machine continuity: reinstalling VS Code would generate a new random ID, orphaning all existing vectors in Pinecone and requiring a full re-index.

The Solution: MD5(git global email) as Stable User ID

The Pinecone namespace is {projectId}::{userId} where userId = MD5(git_email.toLowerCase()). The git email is read from ~/.gitconfig (the global git config file on the machine) using:

git config --global user.email  →  "nawfal@gmail.com"
MD5("nawfal@gmail.com")         →  "b7f2a1c5d3e4f6a7..."

This has exactly the right properties:

Scenariogit emailuserIdOutcome
Same developer, same machineSameSame✓ Reuses existing vectors
Same developer, new machineSame (configured by user)Same✓ Reuses existing vectors
VS Code reinstallSame (lives in ~/.gitconfig)Same✓ Nothing changes
Friend on same repoDifferentDifferent✓ Separate namespace, no conflict
Friend tries to edit your vectorsDifferent emailDifferent namespace✓ Cannot touch your vectors by design

Hashing with MD5 ensures the actual email address is never sent to Pinecone or any external service. The hash is just a stable 32-character fingerprint. The toLowerCase() call ensures Nawfal@gmail.com and nawfal@gmail.com produce the same hash.

If the git email is not configured, the extension shows a clear VS Code error message with the exact command to run (git config --global user.email you@example.com) and refuses to index. There is no silent fallback — a random ID would create a different namespace every time and is worse than showing an error.

C-09 Cross-Machine Portability (Same Developer, New Machine)

The Problem

When the same developer opens the project on a second machine (or clones the repo fresh), there is no .smart-search/ folder (it is gitignored). The index.json is empty, so every function looks "new" — the system does not know which vectors are already in Pinecone from the first machine. A naive implementation would re-embed everything from scratch, wasting the full initial indexing cost.

How It Works in Practice

This is handled gracefully because of the namespace and upsert design:

  1. Developer configures git with the same email on the new machine.
  2. The same userId (MD5 of email) is computed → same Pinecone namespace.
  3. The local index.json is empty → all functions appear new → all get re-embedded.
  4. Pinecone's upsert operation is idempotent: upserting a vector with an existing ID overwrites it, it does not duplicate it. So re-embedding unchanged functions is wasteful but not harmful.
  5. After indexing completes, index.json is populated and future runs are incremental again.
Cost of Cross-Machine Sync
The one-time full re-index on a new machine costs the same as the original initial index. After that, incremental updates resume. The Pinecone vectors are not lost between machines — only the local hash cache needs to be rebuilt.

A future optimization would be to store a copy of index.json in a machine-local location outside the project (e.g. ~/.smart-search/{projectId}.json), so cloning the same repo on a second machine that has indexed it before would skip re-indexing entirely.

C-10 Why Voyage AI, Not OpenAI Embeddings

The Problem

OpenAI provides two general-purpose embedding models: text-embedding-3-small and text-embedding-3-large. Both are trained primarily on internet text (articles, Wikipedia, books, web pages). They work well for natural language similarity tasks but are not optimized for code.

Code has properties that differ fundamentally from natural language:

  • Function and variable names are often abbreviations, acronyms, or camelCase combinations with no standalone meaning (e.g. getUById, pdResp).
  • Punctuation carries semantic meaning ({, :, ->, =>) in a way it does not in English.
  • The relationship between an English query ("validate token") and the code that implements it (if ($t['exp'] < time())) requires learned code-specific associations that a general model does not have.

During early testing using OpenAI embeddings, AI search produced scores in the 35–55% range for clearly relevant results. At the default 35% threshold, too many irrelevant results were returned. Raising the threshold to 50% caused genuine matches to be filtered out.

The Solution: voyage-code-2

Voyage AI's voyage-code-2 model is specifically trained on code + natural language pairs from GitHub, documentation, Stack Overflow, and technical writing. It learns to bridge the semantic gap between English descriptions and code implementations.

FeatureOpenAI text-embedding-3-smallVoyage voyage-code-2
Training focusGeneral internet textCode + NL pairs
Dimensions15361536
Asymmetric modeNoYes (document / query)
Typical code score range35–55%60–85%

Asymmetric Search — document vs query

A unique feature of Voyage AI is asymmetric embedding: code chunks and search queries are embedded with different model modes. The chunker uses input_type="document" when embedding functions for storage. The search pipeline uses input_type="query" when embedding the user's query. These two modes produce vectors that are shaped differently — document vectors are optimized to be retrievable, query vectors are optimized to point toward relevant documents.

In practice this means a natural language query vector, even though it looks nothing like the code syntax, is positioned in the vector space to be close to the relevant function vectors. This is the core mechanism that enables semantic search.

C-11 Minimum Query Length (20 Characters)

The Problem

Short queries produce low-quality embedding vectors. A 2-word query like "get user" embeds to a vector near the center of the semantic space, pointing weakly in many directions at once. The cosine similarity between this vague vector and almost any function vector is in the 30–45% range — just enough to pass the threshold but with no genuine signal. The result is a page of results that appear random.

More fundamentally, Voyage AI's query mode is designed for retrieval of specific information. It performs well when given a full phrase or sentence that articulates intent. Short queries deprive the model of context.

The Solution: 20-Character Minimum in the Frontend

Before sending an AI search request, main.js checks the query length. If it is under 20 characters, a warning is shown and the request is not sent. The user is prompted to describe what the code does rather than just naming a keyword.

20 characters corresponds roughly to 3–4 meaningful words — enough for a short intent phrase:

  • Blocked — 13 chars "validate token"
  • Borderline — 19 chars "validate auth token"
  • Allowed — 26 chars "validate user auth token"
  • Ideal — 45+ chars "check if the user authentication token has expired"

This constraint only applies to AI search. Normal search has no minimum length — searching for a 2-character variable name with normal regex search is perfectly valid.

C-12 Line-Level Result Pinpointing

The Problem

Pinecone's vector search returns results at the function level. A result like "verifyToken(), lines 5–42" tells the developer which function is relevant but requires them to manually scan up to 38 lines to find the specific part that answers their query. For a question like "where do we check token expiry?", the answer might be a single if statement on line 21 inside a 38-line function. The developer should not have to find it manually.

The Solution: GPT Line Locator

After Pinecone returns the matched functions, a second LLM call is made for each result to identify the single most relevant line. The process:

  1. Read the function's lines from disk (using the file path and line numbers from Pinecone metadata).
  2. Build a numbered listing: "21: if ($token['expires_at'] < time()) {".
  3. Send the query + numbered listing to gpt-4o-mini with the instruction to return only JSON: {"line": N, "content": "..."}.
  4. Parse the JSON response. Clamp the returned line number to the valid range [start_line, end_line] (GPT occasionally returns out-of-range values).
# line_locator.py — the prompt sent to GPT
f'Search query: "{query}"\n\n'
f'Code (with line numbers):\n{numbered[:3000]}\n\n'
f'Which single line number is most directly relevant? '
f'Reply ONLY with JSON: {{"line": <number>, "content": "<exact line text>"}}'

The response is parsed with a regex search for {"..."} to handle cases where GPT adds surrounding text before or after the JSON despite the instruction.

When the user clicks a result in the search panel, VS Code opens the file and moves the cursor directly to match_line — the line the GPT identified, not the start of the function.

Fallback: If the file cannot be read, GPT fails, or JSON parsing fails, the result falls back to the function's start_line with empty content. The result is still shown and clickable — just without line-level precision.

C-13 GPT Summary as Semantic Bridge Between English and Code

The Problem

Even with a code-specialized model like voyage-code-2, embedding raw code alone produces a vector that leans toward syntax. Consider this PHP function:

function getProductById($id) {
  // fetch product record
  $result = $this->db->query(
    'SELECT * FROM products WHERE id = ? AND deleted_at IS NULL',
    [$id]
  );
  if (!$result) { error_log("Product $id not found"); return false; }
  return $result[0];
}

A developer searching for "fetch a single product from the database by its ID" would expect this to be a top result. But the code itself contains very little of that natural language. The words "fetch," "single," and "database" do not appear. The code surface form is dominated by PHP syntax (->, $, ?, SELECT). Without a bridge, the cosine similarity between the query vector and this code's vector might be 45% — below a reasonable threshold.

The Solution: Pre-Embedding GPT Summarization

Before embedding, GPT generates a plain English summary that is prepended to the code. The text sent to Voyage AI becomes:

Function: getProductById
Fetches a single product's details from the database by its ID using
a SELECT query with a prepared statement. Returns the first row as an
array or false if the product is not found. Includes error logging.

function getProductById($id) {
  $result = $this->db->query('SELECT * FROM products WHERE id = ?', [$id]);
  ...
}

Now the embedding vector encodes both the English meaning (from the summary) and the code syntax. The cosine similarity against "fetch a single product from the database by its ID" rises to 81% — well above any reasonable threshold.

The GPT prompt specifically asks for:

  1. What the function does and why (purpose, not just mechanics)
  2. Whether it fetches or writes data (SQL, API, file, etc.)
  3. Whether it contains logging, debugging, or error handling

These three dimensions produce summaries that match all the common developer query patterns: "where do we read X from the database?", "where does logging happen?", "find where we call the payment API."

The summary is also stored in Pinecone metadata (summary field, max 500 chars) and displayed in the search result card. This gives the developer an immediate plain English description of the matched function before they click through.

C-14 Parallel API Calls with ThreadPoolExecutor

The Problem

Two stages of the pipeline require multiple LLM calls — one per item:

  • Indexing: summarizing N new/changed functions (one GPT call each)
  • AI Search: running the line locator on up to 10 matched results (one GPT call each)

If made sequentially, summarizing 10 functions would take ~3 500ms (10 × ~350ms per gpt-4o-mini call). Running the line locator on 8 results would take ~2 800ms. These numbers are unacceptable for a developer tool — developers expect sub-second feedback from their editor tools.

The Solution: ThreadPoolExecutor for All LLM Calls

Python's concurrent.futures.ThreadPoolExecutor submits all LLM calls simultaneously in separate threads. Since GPT calls are I/O-bound (waiting for an HTTP response), not CPU-bound, Python's GIL does not limit parallelism — network I/O releases the GIL, allowing all threads to genuinely run concurrently.

# server.py — parallel summarization of N chunks
with ThreadPoolExecutor(max_workers=max(1, len(chunks_to_embed))) as executor:
    # Submit ALL summarize calls at once — they all start simultaneously
    futures = {
        executor.submit(summarize_chunk, chunk): chunk
        for chunk in chunks_to_embed
    }
    # Collect results as each call completes (order doesn't matter)
    for future in as_completed(futures):
        chunk = futures[future]
        try:
            chunk["summary"] = future.result()
        except Exception as e:
            chunk["summary"] = ""  # failure is non-fatal; embed without summary

The same pattern is used for the line locator:

with ThreadPoolExecutor(max_workers=max(1, len(results))) as executor:
    futures = {
        executor.submit(find_relevant_line, query, r, workspace_path): r
        for r in results
    }
    for future in as_completed(futures):
        r = futures[future]
        try:
            loc = future.result()
            r["match_line"]    = loc["line"]
            r["match_content"] = loc["content"]
        except Exception:
            r["match_line"]    = r["start_line"]  # fallback
            r["match_content"] = ""

The worker count is set to max(1, len(items)) — one thread per item. This maximizes concurrency: all N calls start at the same time, and total wall-clock time equals roughly the time of the single slowest call.

Line-locate results8~2 800ms~350ms8×

End Result
Total AI search time (embed query + Pinecone query + line-locate 8 results) consistently completes in under 1 second on a normal internet connection. Indexing 10 new functions after saving a file takes under 500ms.

10 Security and Privacy Design

Several security decisions were made deliberately:

Source Code Exposure

The actual source code of indexed functions is sent to:

  • OpenAI (summarizer.py): The first 3 000 chars of each new/changed function are sent at index time, once per function change. Only the developer's own OpenAI account is used.
  • Voyage AI (embedder.py): The first 8 000 chars of (summary + code) are sent at index time. Only the developer's Voyage AI account is used.
  • Pinecone: Only vectors (lists of floats) and small metadata strings (max 1 000 chars of code preview) are stored. Full source code is not sent to Pinecone.

All API keys live in backend/.env, which is gitignored. They are never committed or shared.

Git Email Hashing

The developer's git email is hashed with MD5 before use as the user ID. The raw email is never sent anywhere — not to Pinecone, not to Voyage AI, not to OpenAI. The MD5 is sufficient as a stable identity fingerprint and is not reversible.

Local Index File

The index.json file contains only MD5 hashes and relative file paths — no code content, no API keys, no embeddings. Even if accidentally exposed, it reveals nothing sensitive.

CORS

The Python backend returns Access-Control-Allow-Origin: *. This is acceptable because the server binds to localhost by default and is not exposed to the internet. In remote deployment mode, the operator should restrict CORS to the VS Code extension origin.

11 Future Work

1. Sliding-Window Chunking for Very Large Functions

Functions longer than 6 000 characters are currently truncated. A better approach is to split them into overlapping windows (e.g. 3 000-char windows with 500-char overlap) and index each window as a separate chunk with a modified ID (file::funcName::chunk_1, ::chunk_2, etc.). This would allow queries to match any part of a large function, not just its opening.

2. Multi-Root Workspace Support

VS Code supports opening multiple folders simultaneously (multi-root workspaces). The extension currently only processes workspaceFolders[0]. Supporting all roots would require per-root namespacing and status tracking.

3. Cross-Project Search

A "search all my projects" mode could query multiple Pinecone namespaces (one per project-id that has been indexed) and merge results. Useful for developers maintaining multiple microservices who want to find similar patterns across repositories.

4. Query Result Caching

Repeated identical queries within a session currently hit the full pipeline. A short TTL in-memory cache (query string → results) would make repeated searches instant. The cache should be invalidated when any indexing operation completes.

5. Streaming Search Results

Currently the UI shows a spinner until all line-locator calls complete. Streaming partial results — showing each result as its line-locator call finishes — would make the UI feel more responsive. Requires a streaming HTTP response or WebSocket connection between the extension and backend.

6. Automatic Model Version Upgrades

Voyage AI periodically releases improved models (voyage-code-3, etc.). Vectors stored in Pinecone using an older model cannot be meaningfully queried by a newer model because the vector spaces are different. An upgrade path would: detect which model was used for stored vectors, trigger a re-index when the configured model changes, and update model metadata in the index.

7. Search History and Favourites

A local history of recent queries (stored in VS Code's globalState) and the ability to pin useful search results would improve daily use. This is a pure frontend feature requiring no backend changes.

8. Support for More File Types

The current indexer covers 12 language families. Adding support for SQL (stored procedures), Solidity (smart contracts), Dart (Flutter), Scala, Haskell, and others would require new chunker functions. The chunker architecture is designed for easy extension — adding a new language is a self-contained function and a case in the switch statement.

12 Conclusion

Smart Search demonstrates that production-quality semantic code search can be built as a solo personal tool, using cloud AI services composed around a VS Code extension. The key insight driving the whole system is that finding code by meaning rather than by keyword requires three technical bridges: a code-specialized embedding model (Voyage AI's voyage-code-2), a plain English semantic bridge (GPT-generated summaries), and line-level precision (a second GPT call that pinpoints the exact line within a matched function).

The engineering challenges were not in the AI itself — the models work well once correctly integrated — but in the surrounding systems: keeping re-indexing cheap (hash-based change detection), handling multi-user isolation correctly (git email namespacing), parsing 11 different language syntaxes reliably (three block-end algorithms), making parallel LLM calls fast enough for a developer tool (ThreadPoolExecutor), and making the system portable across machines and collaborators (relative paths, configurable backend URL).

Each challenge was solved with a targeted, minimal solution. The overall architecture is simple enough to understand in an afternoon but robust enough to handle the daily workflow of a real development project. The result is a tool that meaningfully changes how code navigation works — where the question "where do we validate payment subscription status?" returns a specific file, function, and line number in under one second, without requiring any knowledge of what that function is called.
